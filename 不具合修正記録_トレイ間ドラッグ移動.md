# 不具合修正記録: トレイのアイテムドラッグ&ドロップ

> 関連ドキュメント: [アーキテクチャ v0.1](./アーキテクチャ.md) / [要件定義 v0.1](./要件定義.md)

## 1. 症状（ユーザー報告）

> トレイの中からフォルダを移動させようとしてもフォルダとかファイル単体がドラッグできないのと、
> トレイごと一緒についてきちゃう。

最終的には「別トレイへドロップしても入らない」という症状まで含め、**4段階の異なる原因**が重なっていた。1つ直すたびに次の層の問題が表面化する構造で、都度「直った」→「まだ直っていない」を繰り返した。本書はその全体像と、各段階で何を誤診し、最終的に何が根本原因だったかを記録する。

---

## 2. 問題1: ウィンドウ全体がアイテムと一緒に動いてしまう

### 症状
ファイル/フォルダをドラッグしようとすると、アイテム単体は動かず、トレイパネル（ウィンドウ）全体がカーソルに追従してしまう。

### 誤診・失敗した修正（2回）

トレイパネルは `.borderless` + `.nonactivatingPanel` の `NSPanel`（`TrayPanel`）で、タイトルバーがないため `isMovableByWindowBackground = true` を常時返すことでパネル全体を背景ドラッグで動かせるようにしていた。これがアイテムの `.draggable` と mouseDown を奪い合っていると判断し、次の2つを試したが、いずれも改善しなかった。

1. **`.onHover` でアイテム上かどうかを判定し、`isMovableByWindowBackground` を動的に切り替える**
   失敗理由: SwiftUI の `.onHover` は内部的に `NSTrackingArea` の `.activeInKeyWindow` 相当でしか発火しない。`TrayPanel` は `.nonactivatingPanel` のためほとんどの時間 key window にならず、ホバー判定自体が機能していなかった。

2. **mouseDown 座標とアイテム矩形（`ItemFrameTracker`）を突き合わせて `isMovableByWindowBackground` を動的に切り替える**
   key window 依存は解消したが、依然として改善しなかった。調査の結果、macOS のウィンドウ背景ドラッグの可否は `NSWindow.isMovableByWindowBackground` そのものではなく、実際に mouseDown を受け取った `NSView` 側の `mouseDownCanMoveWindow` が個別に判定しており、かつ **SwiftUI 側のジェスチャ処理とウィンドウ移動処理は「どちらか一方が勝つ」のではなく同時に走り得る**ことが判明した。ウィンドウ側のプロパティをいくら動的に切り替えても、`NSHostingView`（SwiftUI 全体を包む1枚の view）側の判定タイミング次第で反映されなかった。

### 根本原因と最終的な修正

`NSHostingView` をサブクラス化した `NonMovableHostingView` を導入し、`mouseDownCanMoveWindow` を常に `false` 固定にした。これにより SwiftUI コンテンツ上の mouseDown が AppKit 標準のウィンドウ移動を発火させる余地を完全に断った。

その代わり、トレイ自体をヘッダー/背景ドラッグで動かす機能は失われるため、`TrayPanel.sendEvent` 内で mouseDown/mouseDragged/mouseUp を自前追跡し、`setFrameOrigin` で直接移動させる完全自前実装に置き換えた（アイテム上・リサイズグリップ上では発動しないよう `ItemFrameTracker` で除外）。

**該当ファイル**: `DesktopTray/Windows/PanelSubclasses.swift`（`TrayPanel`, `ItemFrameTracker`）, `DesktopTray/Windows/TrayWindowController.swift`（`NonMovableHostingView`）

## 3. 問題2: スマートトレイからの移動がバックエンドで握りつぶされる

### 症状
問題1解決後、アイテム自体はつかめるようになったが、別トレイへ移動できないケースがあった。

### 根本原因

`TrayEngine.moveBetweenTrays` は移動元トレイが `.manual` 型であることを前提にしていた。しかしユーザーが操作していたのは「未分類」のような **スマートトレイ**（`SmartTrayEvaluator` によるルール評価でメンバーシップが都度計算されるトレイ。実体としての所属配列を持たない）。移動元がスマートトレイの場合、`moveBetweenTrays` の guard が即座に失敗し、何も起きずに終わっていた。

### 修正

移動元が手動トレイでない（＝スマートトレイ）場合は「削除」ではなく「移動先の手動トレイへの追加」として扱うよう `moveBetweenTrays` を修正。スマートトレイ側の表示内容は次回のルール再評価で自動的に更新される（例:「未分類」ルールは手動未所属のアイテムのみを対象とするため、追加後は自動的にそちらの一覧から外れる）。

**該当ファイル**: `DesktopTray/Domain/TrayEngine.swift`

## 4. 問題2': スマートトレイのアイテムIDが再評価のたびに変わる

問題2の修正と同時に発見した関連バグ。`SmartTrayEvaluator.evaluate()` はルール評価のたびに `TrayItem` を新規生成するが、`id` はデフォルトのランダム `UUID()` だった。ファイル監視（`FileSystemWatcher`）による再評価がドラッグ中に走ると、ドラッグ開始時に掴んだ `itemID`（`TrayItemTransfer.itemID`）がドロップ時には別物になり、たとえ問題2を直しても照合に失敗するケースがあり得た。

### 修正

`TrayItem.stableID(for:)` を追加し、URL の MD5 ハッシュから決定論的な `UUID` を生成。`SmartTrayEvaluator` はこれを使うことで、同じファイルには常に同じ `id` が振られるようにした。

**該当ファイル**: `DesktopTray/Domain/Models/TrayItem.swift`, `DesktopTray/Domain/SmartTrayEvaluator.swift`

## 5. 問題3: 空トレイにドロップ受付エリアが存在しない

### 症状
問題2系を直しても、まだ移動できないケースが残っていた。

### 根本原因

SwiftUI の `.dropDestination(for: TrayItemTransfer.self)` を `LazyVGrid`（アイコンが並ぶグリッドそのもの）にのみ付与していた。永続化データ（`trays.json`）を確認すると、ほとんどの手動トレイが空（`"items": []`）であり、**空の `LazyVGrid` は高さがほぼゼロに潰れる**。結果、移動先トレイに実質ドロップ受付エリアが存在せず、フォルダを離してもどのドロップターゲットにも当たらなかった。

### 修正（暫定）

`.dropDestination` をパネル全域（ヘッダー含む root view）に付け替え、最小高さも確保。この時点ではまだ根本解決ではなかった（次項）。

## 6. 問題4（根本原因）: クロスウィンドウの SwiftUI D&D は信頼できない

### 症状
問題3の修正後もなお、別トレイへドロップしても入らない。

### 根本原因

**各トレイは独立した `NSPanel`（別ウィンドウ）であり、それぞれが自分の `NSHostingView` に SwiftUI ツリーを載せている。** SwiftUI の `.dropDestination` は、**別ウィンドウ（別 `NSHostingView`）で起動した `.draggable` のドラッグを確実に受け取れない**。これはクロスウィンドウ D&D 特有の信頼性問題であり、同一ウィンドウ内（同一トレイ内の並び替え）では問題にならないため、これまでの調査で見過ごされていた。

一方、コードベースには **Finder からのファイルドロップ（`FileDropView`）が既に別ウィンドウ跨ぎで正常に動作していた**。これは SwiftUI の宣言的 D&D API を使わず、`NSView` + `registerForDraggedTypes` + `NSDraggingDestination` というネイティブ AppKit の仕組みを直接使っていたためである。

### 最終修正: ネイティブ D&D 方式への置き換え

`FileDropView` と同じ設計思想で、トレイアイテムの D&D を SwiftUI の `.draggable`/`.dropDestination` から完全に切り離し、ネイティブ実装に置き換えた。

- **`TrayItemDragCoordinator`**（新規, `DragSessionBridge.swift`）: アプリ内シングルトン。全トレイが同一プロセスであることを利用し、ドラッグ中のペイロード（`itemID` / `sourceTrayID`）をペーストボードにシリアライズせず直接保持する。
- **`TrayItemView`**: `.draggable(...)` → `.onDrag { ... }` に変更。ドラッグ開始時に `TrayItemDragCoordinator.shared.begin(...)` を呼び、アプリ内専用のペーストボード型（`com.desktoptray.tray-item`）のみを登録した `NSItemProvider` を返す。
- **`TrayItemDropView`**（新規, `DragSessionBridge.swift`）: `FileDropView` を鏡写しにした `NSView` サブクラス。`registerForDraggedTypes([.trayItem])` で型登録し、`performDragOperation` で `TrayItemDragCoordinator.shared.current` からペイロードを読み取って、同一トレイなら並び替え、別トレイなら移動のコールバックを発火する。`hitTest` は `FileDropView` 同様 `nil` を返し、下層の SwiftUI ボタン等のクリックを妨げない。
- **`TrayWindowController`**: `installGlassContentView` で `TrayItemDropView` をパネルのコンテナ階層（`NSHostingView` の直後、リサイズグリップの手前）に追加し、パネル全域を覆わせる。
- **`OverlayWindowManager` → `AppCoordinator`**: 並び替え/移動のコールバックを中継し、既存のバックエンド（`reorderItem`/`moveItem` → `TrayEngine`）にそのまま接続。
- **`TrayPanelView` / `TrayPanelContainer`**: 不要になった `.dropDestination`/`handleDrop`/`nearestIndex` と、それに伴う未使用クロージャ（`onReorder`/`onMoveFromOtherTray`）を削除。

並び替え位置（挿入インデックス）の計算も、これまでのハードコードされた行/列幅の目算（`nearestIndex`）から、既存の `ItemFrameTracker`（アイテムの実際の表示矩形）を使った判定に置き換え、精度も向上した。

**該当ファイル**: `DesktopTray/Windows/DragSessionBridge.swift`, `DesktopTray/Features/TrayPanel/TrayItemView.swift`, `DesktopTray/Windows/TrayWindowController.swift`, `DesktopTray/Windows/OverlayWindowManager.swift`, `DesktopTray/App/AppCoordinator.swift`, `DesktopTray/Features/TrayPanel/TrayPanelView.swift`, `DesktopTray/Features/TrayPanel/TrayPanelContainer.swift`

## 7. 教訓

- **各トレイが別ウィンドウである**という前提を早い段階で織り込めていれば、SwiftUI 標準の D&D API に固執せず、もっと早くネイティブ方式に切り替えられた。マルチウィンドウ構成では「同一ウィンドウ内では動くが別ウィンドウでは動かない」パターンを疑うべき。
- **AppKit の非公開・半公開挙動（`isMovableByWindowBackground` の実際の判定タイミング）に頼った修正は再現性が低い。** 迷ったら公式に文書化された明確な API（`mouseDownCanMoveWindow` の override、`NSDraggingDestination` の明示的な実装）に寄せるほうが確実。
- **同じバグに見えても実際には複数の独立した原因が積み重なっていることがある。** 1つ直して症状が変化しない/別の形で再現する場合は「直っていない」と早合点せず、直った部分と直っていない部分を切り分けること。
- 既存コードベースに **既に動いている類似実装（`FileDropView`）** があれば、車輪の再発明をせずそれを鏡写しにするのが最も確実な近道だった。
